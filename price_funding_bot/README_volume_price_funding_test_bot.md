## Что делает бот

Бот отслеживает данные BingX по перпетуалам и ищет сценарий **«выдавливание лонгов»**: отрицательный фандинг усиливается (по модулю растёт), при этом цена растёт за выбранное окно времени.

Если условия выполняются, бот отправляет сообщение в Telegram с найденными тикерами.

Калибровка порогов по файлу `prices.csv`: там 3-минутные срезы, **минута 0** — момент, который заказчик пометил как «нужный» для раннего сигнала. Под эти примеры подстроены значения по умолчанию (окно **15 минут** ≈ пять 3m-свечей до нуля, опрос **каждые 3 минуты**).

## Ответ заказчику: «какой диапазон цены и как фильтруется?»

**Это не диапазон абсолютной цены** (не «от X$ до Y$»).

Фильтр по цене — это **диапазон допустимого процентного роста** за окно `lookback_minutes`:


\text{pricegrowth} = \frac{P_{\text{now}} - P_{\text{past}}}}{P_{\text{past}}} \times 100


где P_{\text{now}} — текущая цена (mark, иначе index, иначе last с API), P_{\text{past}} — цена в **ближайшем сохранённом срезе не новее**, чем `lookback_minutes` назад.

Сигнал проходит, если одновременно:

- `price_pump_min <= price_growth_% <= price_pump_max`

То есть «диапазон цены» в настройках — это **минимальный и максимальный допустимый % роста** за выбранный интервал (например 0.25–12% при дефолтах после калибровки по CSV).

## Логика расчёта сигнала

По каждой монете бот хранит историю точек `(timestamp, funding_%, price)` на каждом опросе.

### 1. Фандинг (уровень)

Текущий funding в процентах: `lastFundingRate` из API BingX умножается на 100 (как в UI «%»).

Условие «в коридоре»:

- `funding_range_min <= funding_now <= funding_range_max`

По умолчанию после калибровки по `prices.csv`: от **−0.60%** до **−0.005%** (слабый минус у входа и более глубокий минус у других кейсов).

### 2. Усиление минусового фандинга за окно lookback

Берётся прошлый срез (f_{\text{past}}, P_{\text{past}}) — та же логика «не новее N минут назад», что и для цены.

Проверка по **росту модуля** (насколько фандинг «углубился» в минус):


|f_{\text{now}}| - |f_{\text{past}}| \ge \texttt{deltafundingdrop}


`delta_funding_drop` задаётся в **процентных пунктах** того же масштаба, что и funding в сообщениях (например `0.0005` — на полтысячного процента по модулю за 15 мин в данных Image_8).

### 3. Рост цены за то же окно

Тот же `price_growth_%`, что в формуле выше; границы — `price_pump_min` / `price_pump_max`.

### 4. Частота и окно

- Опрос: каждые **180 секунд** (3 минуты), чтобы быть ближе к 3m-графикам из CSV.
- `**lookback_minutes`** (по умолчанию **15**): и цена, и funding сравниваются с прошлым срезом не новее этого времени. Команда: `/fundbot_lookback_minutes`.

После старта бота первые сигналы возможны только когда для символа накопилась точка истории **старше** текущего момента минимум на `lookback_minutes` (иначе в debug будет `skip:no_lookback_sample`).

## Параметры по умолчанию (после `prices.csv`)


| Параметр                    | Значение      | Смысл                           |
| --------------------------- | ------------- | ------------------------------- |
| `lookback_minutes`          | 15            | Окно сравнения с прошлым срезом |
| `funding_range_min` / `max` | −0.6 / −0.005 | Допустимый текущий funding, %   |
| `delta_funding_drop`        | 0.0005        | Мин. прирост |f| за окно, п.п.  |
| `price_pump_min` / `max`    | 0.25 / 12.0   | Допустимый % роста цены за окно |


Их можно менять командами ниже — под другие рынки имеет смысл ужесточить коридор funding и/или поднять `delta_funding_drop`.

## Anti-spam по повторным сигналам

Эффективная пауза по одному тикеру (минуты):

`max(mention_cooldown_min, repeat_signal_cooldown_min)`

- `**mention_cooldown_min`** — ручной cooldown (может быть `0`).
- `**repeat_signal_cooldown_min**` (по умолчанию **45**) — защита от повторных сообщений, пока условия совпадения держатся (как несколько алертов подряд по 0G-USDT).

Полностью убрать паузу: и `mention_cooldown_min = 0`, и `/fundbot_repeat_signal_cooldown 0`.

## Команды Telegram

- `/start` — краткая справка по командам.
- `/fundbot_connect` — привязать текущий чат/топик для алертов.
- `/fundbot_status` — показать текущие настройки.
- `/fundbot_short_trap on|off` — включить/выключить сигналы «выдавливание лонгов» (в конфиге ключ `short_trap_enabled`).
- `/fundbot_funding_range <min> <max>` — рабочий диапазон текущего funding (например `-0.2 -0.01`).
- `/fundbot_funding_threshold <число>` — legacy: верхняя граница funding и узкое окно ниже неё.
- `/fundbot_delta_funding <число>` — минимальный прирост funding за окно lookback (п.п.).
- `/fundbot_price_pump_min` / `fundbot_price_pump_max` — минимальный и максимальный **процент роста цены** за окно lookback.
- `/fundbot_lookback_minutes <минуты>` — окно сравнения (3–120), по умолчанию 15.
- `/fundbot_mention_cooldown <минуты>` — пауза на повторные упоминания тикера.
  - Пример: `/fundbot_mention_cooldown 30`
  - Выключить: `/fundbot_mention_cooldown off` или `/fundbot_mention_cooldown 0`
- `/fundbot_repeat_signal_cooldown <минуты>` — минимальный интервал между **сигналами** по одному тикеру (анти-спам; по умолчанию 45). `0` — выключить этот пол.

## Что видно в сообщении Telegram

Для каждой монеты бот показывает:

- тикер (ссылка на BingX);
- текущий funding;
- funding на момент среза **lookback** минут назад;
- изменение funding и рост f относительно порога;
- рост цены за **lookback** минут и допустимый коридор в процентах;
- примерно сколько минут до следующего funding (если доступно).

## Частота работы

- Проверка сценария «выдавливание лонгов»: каждые **3 минуты**.
- Часовой процесс (дополнительный цикл): по минуте `alert_minute` в UTC.

## Debug-файлы по каждому проходу

Бот сохраняет диагностические файлы в папку `debug_short_trap` при каждом цикле проверки.

На каждый проход создаются файлы:

- `*_00_overview.txt` — время парса, `lookback_minutes`, интервал опроса и пороги.
- `*_01_all_coins.tsv` — полный список монет с funding, ценой и статусом (почему монета пропущена).
- `*_02_funding_filter.tsv` — этап фильтра funding: коридор и прирост f за окно.
- `*_03_price_filter.tsv` — этап фильтра цены: % роста за окно.
- `*_04_alerts.tsv` — итоговые монеты, попавшие в сигнал.

Файлы автоматически ротируются: хранятся только последние ~240 файлов, старые удаляются.
